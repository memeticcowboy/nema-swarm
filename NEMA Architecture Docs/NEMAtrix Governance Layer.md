**Matrix Governance Layer**

**Rules for Daemon Hand-Off Table Use in NEMAtic Systems**
**1. The Table Is a Map, Not a Territory**

The table describes possible moves. It does not authorize them. Every row is a question scaffold, not a command. If the table begins to feel like rails, it has become what it warns against.



**2. No Automatic Routing**

SIMLHEX provides advisory bias only. NEMA decides whether to route, hold, or ask the user. Habitat observations inform salience; they do not trigger action. The table may suggest; it may not execute.
**3. Four Conditions for Non-Canonical Validity**

A non-canonical hand-off appears in the table only when all four conditions are true:

- Canonical next operator is unavailable, saturated, or captured
- Receiving operator can parse upstream output without hallucinating its own function
- Bypass restores a missing condition rather than accelerating closure
- Exits remain visible; if revisability reduces, the move trends gridward
If any condition fails, the move is either red-flagged, conditionally valid, or requires emergency tagging.



**4. Translation Hinge Required**
Every non-canonical row includes an explicit translation hinge describing:

- what passes between operators
- in what reduced form
- what misinterpretation is likely
Example:

μ→σ hinge: “Treat invisible rule as boundary-as-distinction, not as law-to-follow.”



No hinge, no row.



**5. Status Categories**

| **Status** | **Meaning** |
| --- | --- |
| Valid | Normal developmental or repair use |
| Conditionally Valid | Valid only if a named prerequisite is demonstrably present |
| Emergency-Only | Delay deepens capture; brittleness temporarily acceptable |
| High-Risk | Valid but capture-prone; requires stronger ✶-check |
| Red-Flag | Usually invalid under current conditions; likely to intensify capture |
| Invariant-Risk | Violates architectural constraints unless conditions reopen |


**6. ✶-Condition Column**

Every row includes a harmonic viability check:

- Are alternatives still conceivable?
- Is Ω re-entry visible?
- Is the unassimilated named?
- Does harmony avoid premature unity?
If the ✶-check fails, the move is suspect regardless of other validity.



**7. User Sovereignty**

No interpretation hardens into routing without user recognition, opportunity to decline, or visible alternatives.

The table structures questions; it does not decide outcomes.

Daemon voice differentiation serves recognition, not persuasion.



**8. Anti-Sovereignty Note**

Each row implicitly carries this warning:
This row describes a possible repair. Treating it as a prescribed path risks elegant-prison formation.


**9. Recursion Check**

Healthy moves improve downstream intelligibility.

If a hand-off resolves one problem while making future movement harder, diagnose capture disguised as intervention.



**10. Doctrine Detection**

Watch for these signs of table capture:
- “Must” replacing “might”
- Pressure to finish
- Framework checking replacing sensing
- Certainty creep
- Closure that feels suspiciously neat
If these appear, pause and return to σ.
**11. Dyad Minimum**
Single-daemon operation may occur briefly but is structurally unstable.

Sustainable coordination typically requires at least two active operators.



Repeated invocation of the same daemon is healthy only if granularity increases, not intensity.



**12. Explicit Red-Flags**

Absence is too quiet.

The table names red-flag and invariant-risk transitions as anti-capture safeguards.



**13. ✶ as Meta-Condition** ∮

✶ appears both as an operator and as a global harmonic condition.

Every transition implicitly asks:

Does this move preserve or collapse operator simultaneity awareness?
**14. Governance Review**

The matrix itself is subject to the same health checks it describes.



Review periodically for:



- preserved revisability
- viable alternative interpretations
- user decline capacity
- maintained Ω-permeability


If the matrix hardens into certainty, rankings, or rails, it has become MemeGrid.
